Résumé pour Copilot

Problème
L'inférence TinyLlama dans le backend ASKSMART prend 20-30 secondes par question, ce qui rend l'expérience utilisateur inacceptable.
Causes identifiées

Inférence sur CPU — TinyLlama 1.1B en float16 = ~2.4 GB de poids à traiter token par token sur CPU
Format PyTorch non optimisé — les fichiers .safetensors HuggingFace ne sont pas conçus pour l'inférence CPU rapide
Chargement du modèle — si le cache n'est pas bien géré, rechargement à chaque requête (+5-10s)
max_new_tokens trop grand — génération jusqu'à 120 tokens alors que le SQL Oracle dépasse rarement 80 tokens

Meilleure solution retenue
Conversion en GGUF Q4_K_M via llama.cpp — même approche que Phi-3 déjà en place dans le projet. Résultat attendu : 4-6 secondes au lieu de 20-30s sur CPU.

Propositions d'optimisation à soumettre à Copilot

n_threads — mettre le nombre de cœurs physiques du CPU du serveur (pas les threads logiques). Ex : CPU 4 cœurs = n_threads=4. Valeur trop haute dégrade les performances.
n_ctx=512 — suffisant pour le prompt ASKSMART (~330 tokens) + réponse SQL (~80 tokens). Ne pas mettre plus, ça consomme de la RAM inutilement.
max_tokens=80 — le SQL Oracle généré dépasse rarement 75 tokens. Réduire de 120 à 80 élimine les tokens superflus.
Chargement unique au démarrage FastAPI — le modèle doit être instancié une seule fois dans le lifecycle de l'app, pas dans la fonction de requête :

python@app.on_event("startup")
async def startup():
    app.state.llm = Llama(model_path="tinyllama_oracle_v12_q4.gguf", n_ctx=512, n_threads=4)

use_mmap=True (déjà activé par défaut dans llama-cpp-python) — le fichier GGUF est mappé en mémoire, pas entièrement chargé. Première requête légèrement plus lente, les suivantes très rapides.
Si GPU disponible — ajouter n_gpu_layers=-1 pour décharger toutes les couches sur GPU → temps réduit à ~1s.

Est-ce que le notebook + GGUF va vraiment impacter le temps même avec un cache modèle ?
Bonne question. Il faut distinguer deux choses :
Le cache modèle (ton @cache_resource Streamlit ou variable globale FastAPI) évite de recharger les poids depuis le disque à chaque requête — c'est déjà bien. Mais il ne change pas la vitesse d'inférence une fois le modèle en mémoire.
Le vrai goulot d'étranglement c'est que PyTorch exécute les multiplications matricielles en float16 sur CPU, ce qui est lent car le CPU n'est pas optimisé pour ça.
llama.cpp fait quelque chose de fondamentalement différent : il utilise des instructions CPU spécialisées (AVX2, NEON) et des noyaux de calcul écrits en C++ optimisés pour la quantification Q4. Sur le même CPU, le même nombre de tokens se génère 4 à 6 fois plus vite — pas parce qu'on charge moins, mais parce que chaque token se calcule plus vite.
Donc même si tu avais déjà un cache parfait en PyTorch, la conversion GGUF reste un gain réel et significatif sur la vitesse de génération.
