### Nouvelles tables (5 ajouts majeurs)

| Table | Pourquoi elle existe | N'existait pas en v2.0 car… |
|-------|---------------------|----------------------------|
| `provider_tokens` | Cache les tokens OAuth des prestataires pour éviter de redemander un token à chaque appel API | En v2.0 il n'y avait pas encore d'intégration OAuth MoneyGram |
| `quotes` | Stocke les devis (frais + taux de change) retournés par le prestataire avant de confirmer la transaction | En v2.0 les frais étaient calculés uniquement localement |
| `refunds` | Gère les remboursements via l'API prestataire (MoneyGram Refund API) | En v2.0 il n'y avait aucune gestion des remboursements |
| `service_options` | Définit les modes de service : cash, virement bancaire, wallet mobile, carte | En v2.0 on ne supportait que le cash |
| `status_history` | Garde l'historique de chaque changement de statut d'un dépôt/retrait/remboursement | En v2.0 on ne gardait que le statut courant |

## 📋 VUE D'ENSEMBLE

### **Objectif**
Développer le backend de l'application SMART Transfer — une plateforme de transfert d'argent pour caissiers en agence avec support multi-prestataires (MoneyGram, NAFA, Ria, Kori, Capital Service, Saaka, Transfast).

### **Périmètre**
- API REST FastAPI (Python 3.11.9)
- Base de données MySQL 8.0 (**19 tables, nommées en anglais**)
- Authentification JWT (HttpOnly Cookie) via identifiant/mot de passe (local ou Active Directory)
- Intégration prestataires alignée sur l'API MoneyGram (flux Quote → Update → Commit)
- Système de frais hybride (estimation locale + frais réels du prestataire)
- Tests pytest (coverage 80%+)
- Déploiement PM2 sur smart2d-dev

### **Décision de langue (v3.0)**

> **Pourquoi ce changement ?**
>
> En v2.0 tout était en français (tables, colonnes, ENUM, code Python, endpoints).
> Après l'audit MoneyGram, on a constaté que cela créait un mapping permanent
> entre les données API (anglais) et notre BDD (français), source de bugs
> et de confusion.
>
> **Règle v3.0 :**
> - 🇬🇧 **Anglais** : Base de données (tables, colonnes, ENUM), code Python
>   (classes, fonctions, variables), endpoints API, fichiers Python
> - 🇫🇷 **Français** : Documentation (ce document et les autres), commentaires
>   dans le code, interface utilisateur, messages affichés aux caissiers/clients

## 🎯 DÉCISIONS TECHNIQUES CLÉS

### **Base de données — v3.0**
- **19 tables** nommées en anglais (`deposits`, `withdrawals`, `cashiers`, `providers`, `quotes`, etc.)
- **ENUM en anglais** : `pending`, `completed`, `failed`, `cancelled`, `available`, `in_transit`, etc.
- **11 statuts de dépôt** alignés sur l'API MoneyGram
- **5 nouvelles tables majeures** : `provider_tokens`, `quotes`, `refunds`, `service_options`, `status_history`
- **Modèle de frais hybride** : estimation locale (`fee_configurations`) + frais réels API (`deposits.provider_fee` + `deposits.local_fee`)
- **Flux de transaction multi-étapes** : Quote → Update → Commit
- **Adresse structurée** sur `customers`
- **`date_of_birth`** sur `customers` : obligatoire pour MoneyGram
- **Suivi statut prestataire** : `provider_status` + `provider_sub_status` sur `deposits`/`withdrawals`

### **Authentification**
- **JWT avec HttpOnly Cookie**
- Connexion par **`employee_id`** + `password`
- Mode **local** ou **Active Directory**
- Cookie name : `access_token`

### **Intégration MoneyGram**
- **OAuth 2.0** client_credentials → `provider_tokens`
- **Transfer** : quote → update → commit
- **Payout** : retrieve → update → commit
- **Refund** : retrieve → commit
- **Status** : par `transactionId` ou `referenceNumber`
- **Gestion erreurs** : `category`, `code`, `message`, `offendingFields`

### **Tests**
- **pytest** : 80% coverage minimum
- Données de test : Faker (`fr_FR`) + factory_boy
- Mocks : uniquement pour APIs externes

## ❓ FAQ RAPIDE

### **Q: Pourquoi tout n'est plus en français ?**
**R:** Parce que les APIs prestataires (MoneyGram, Ria…) parlent anglais. Garder des tables françaises forçait un mapping permanent et fragile. Le code/BDD passent en anglais, mais toute la documentation et l'interface restent en français.

### **Q: Quelles sont les 5 nouvelles tables principales ?**
**R:** `provider_tokens`, `quotes`, `refunds`, `service_options`, `status_history`.

### **Q: C'est quoi le système de frais hybride ?**
**R:** `fee_configurations` = estimation locale. L'API prestataire = frais réels. Le client paie `provider_fee + local_fee`.

### **Q: C'est quoi le flux multi-étapes ?**
**R:** Quote → Update → Commit. MoneyGram l'exige pour une intégration correcte.
