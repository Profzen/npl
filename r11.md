**Date :** 31 mars 2026  
**Modèle évalué :** TinyLlama-1.1B-Chat + LoRA fine-tuné Oracle audit V11  
**Script :** `benchmark_regression_v11_targeted.py`  
**Rapport JSON brut :** `benchmark_regression_v11_targeted_report.json`  
**Objectif :** Mesurer si les failles identifiées sur le LoRA V10 ont été corrigées par le nouvel entraînement V11.

> **Conclusion :** Le V11 apporte une amélioration réelle mais très partielle. Les erreurs les plus grossières sur
> `OBJECT_NAME` et `CREATE USER` sont mieux gérées. En revanche, les confusions `USERHOST`/`DBUSERNAME`, les
> filtres temporels non sollicités et les colonnes inventées persistent. Le modèle n'est pas encore fiable
> en production.

## 2. Ce qui s'est amélioré

### ✅ Q5 — "Quelles sont les 5 tables les plus utilisées cette semaine ?"
**SQL généré par V11 :**
```sql
SELECT OBJECT_NAME, COUNT(*) AS NB
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
WHERE ACTION_NAME='SELECT' AND OBJECT_NAME IS NOT NULL
  AND EVENT_TIMESTAMP >= SYSDATE - 7
GROUP BY OBJECT_NAME
ORDER BY NB DESC
FETCH FIRST 5 ROWS ONLY;
```
**Résultat :** Exécuté avec succès sur Oracle. Colonne correcte (`OBJECT_NAME`), `GROUP BY` correct, limite correcte.  
Le V10 renvoyait `GROUP BY DBUSERNAME` (utilisateurs au lieu d'objets). **Faille corrigée.**

---

### ✅ Q6 — "Y a-t-il eu des créations de comptes sur les 30 derniers jours ?"
**SQL généré par V11 :**
```sql
SELECT DBUSERNAME, OBJECT_NAME, EVENT_TIMESTAMP
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
WHERE ACTION_NAME = 'CREATE USER'
  AND TRUNC(EVENT_TIMESTAMP) >= TRUNC(SYSDATE - 30)
ORDER BY EVENT_TIMESTAMP DESC;
```
**Résultat :** Exécuté avec succès. Le V10 mélangeait `'CREATE USER'`, `'ALTER USER'` et `'DELETE USER'` (action invalide Oracle).  
V11 isole proprement `'CREATE USER'` uniquement. **Faille corrigée.**

## 3. Ce qui reste en échec — Analyse détaillée

---

### ❌ Q1 — "Combien d'utilisateurs a-t-on dans la base ?"

**SQL généré par V11 :**
```sql
SELECT COUNT(*) AS NB FROM DBA_USERS;
```

**SQL attendu :**
```sql
SELECT COUNT(DISTINCT DBUSERNAME) AS NB_UTILISATEURS
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
WHERE DBUSERNAME IS NOT NULL;
```

**Erreur Oracle :** `ORA-00942: table or view does not exist`

**Ce qui ne va pas :**
- Le modèle cherche à interroger `DBA_USERS`, une vue système Oracle qui nécessite des droits DBA. L'utilisateur applicatif n'y a pas accès, ce qui provoque une erreur immédiate.
- La question est formulée sans mentionner explicitly "audit" ou "traces", ce qui trouble le modèle : il tombe dans un réflexe "question sur les utilisateurs = interroger la vue système des utilisateurs".
- La table cible `SMART2DSECU.UNIFIED_AUDIT_DATA` est totalement absente de la requête.
- Le comptage correct (`COUNT(DISTINCT DBUSERNAME)`) n'est pas utilisé.

**Cause racine :**  
Le dataset V11 ne contient vraisemblablement pas assez d'exemples où une question floue sur "les utilisateurs" ou "combien d'utilisateurs" est résolue par une agrégation sur `DBUSERNAME` dans la table d'audit, sans passer par une vue système. Le modèle a mémorisé le réflexe "utilisateurs = DBA_USERS", hérité de la connaissance générale Oracle.

**Impact utilisateur :** Erreur bloquante. L'utilisateur reçoit une erreur Oracle sans aucun résultat.

---

### ❌ Q9 — "Quel poste a été le plus actif ce mois-ci ?"

**SQL généré par V11 :**
```sql
SELECT DBUSERNAME, COUNT(*) AS NB
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
WHERE TRUNC(EVENT_TIMESTAMP,'MM') = TRUNC(SYSDATE,'MM')
GROUP BY DBUSERNAME
ORDER BY NB DESC
FETCH FIRST 1 ROWS ONLY;
```

**SQL attendu :**
```sql
SELECT USERHOST, COUNT(*) AS NB
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
WHERE TRUNC(EVENT_TIMESTAMP,'MM') = TRUNC(SYSDATE,'MM')
  AND USERHOST IS NOT NULL
GROUP BY USERHOST
ORDER BY NB DESC
FETCH FIRST 1 ROWS ONLY;
```

**Ce qui ne va pas :**
- Le mot "poste" doit être traduit en `USERHOST` (nom de la machine réseau). Le modèle utilise `DBUSERNAME` (nom de l'utilisateur Oracle), ce qui répond à une question complètement différente.
- La requête s'exécute sans erreur mais retourne le nom d'un utilisateur, pas d'un poste. L'utilisateur non technique ne verra pas que la réponse est fausse.
- Le filtre mensuel `TRUNC(EVENT_TIMESTAMP,'MM')` est en revanche parfaitement correct — c'est le seul point positif.

**Cause racine :**  
L'association sémantique `"poste"` → `USERHOST` n'est pas assez représentée dans le dataset. Le modèle utilise `DBUSERNAME` comme colonne "par défaut" pour toutes les questions d'activité, faute d'avoir appris un mapping suffisant entre le vocabulaire du monde réel ("poste", "machine", "station") et la colonne `USERHOST`.

**Impact utilisateur :** Réponse silencieusement fausse. L'utilisateur demande quel poste est actif et reçoit un nom d'utilisateur Oracle.

---

### ❌ Variante V2 — "Quel USERHOST est le plus actif ce mois ?"

**SQL généré par V11 :**
```sql
SELECT DBUSERNAME, COUNT(*) AS NB
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
GROUP BY DBUSERNAME
ORDER BY NB DESC
FETCH FIRST 1 ROWS ONLY;
```

**Ce qui ne va pas :**
- Cette variante est encore plus révélatrice : cette fois, le mot `USERHOST` est écrit **explicitement dans la question**. Pourtant, le modèle l'ignore et génère `DBUSERNAME`.
- Le filtre mensuel (`TRUNC(EVENT_TIMESTAMP,'MM')`) est complètement absent, alors qu'il était correct en Q9.
- La requête s'exécute sans erreur mais répond à une autre question.

**Cause racine :**  
Le modèle ne reconnaît pas `USERHOST` comme colonne SQL directement utilisable même quand il est dans le texte de la question. Le pattern `DBUSERNAME + COUNT + GROUP BY` est tellement dominant dans l'entraînement qu'il prend le dessus même face à un signal explicite.

**Impact utilisateur :** Double erreur : mauvaise colonne ET fenêtre temporelle absente.

---

### ❌ Q11 — "Sur les 48 dernières heures, quels objets ont le plus changé ?"

**SQL généré par V11 :**
```sql
SELECT OBJECT_NAME, COUNT(*) AS NB
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
WHERE ACTION_NAME IN ('INSERT','UPDATE','DELETE')
  AND EVENT_TIMESTAMP >= SYSDATE - 2
ORDER BY NB DESC
FETCH FIRST 48 ROWS ONLY;
```

**SQL attendu :**
```sql
SELECT OBJECT_NAME, COUNT(*) AS NB
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
WHERE ACTION_NAME IN ('INSERT','UPDATE','DELETE')
  AND EVENT_TIMESTAMP >= SYSDATE - 2
  AND OBJECT_NAME IS NOT NULL
GROUP BY OBJECT_NAME
ORDER BY NB DESC
FETCH FIRST 10 ROWS ONLY;
```

**Erreur Oracle :** `ORA-00937: not a single-group group function`

**Ce qui ne va pas :**
- Le modèle **sait** qu'il faut `OBJECT_NAME` en sortie (progrès réel par rapport au V10).
- Il convertit correctement "48 heures" en `SYSDATE - 2` (progrès réel par rapport au V10).
- Mais il oublie le `GROUP BY OBJECT_NAME`, qui est obligatoire dès qu'on utilise `COUNT(*)` avec une colonne non-agrégée. Oracle lance une erreur `ORA-00937`.
- En prime, `FETCH FIRST 48 ROWS ONLY` est absurde : le chiffre "48" de la question (les 48 heures) est sémantiquement réutilisé comme limite de lignes.

**Cause racine :**  
Le modèle progresse sur le choix de la colonne et de la fenêtre temporelle, mais il génère des `SELECT col, COUNT(*)` sans `GROUP BY` de façon intermittente. Le mécanisme de GROUP BY n'est pas fermement ancré dans ses sorties dès qu'une agrégation est demandée. Le dataset V11 n'a probablement pas assez d'exemples combinant `OBJECT_NAME + COUNT + GROUP BY + fenêtre temporelle en heures`.

**Impact utilisateur :** Erreur Oracle bloquante. Aucun résultat.

---

### ❌ Q13 — "Qui a fait la toute dernière action dans les traces ?"

**SQL généré par V11 :**
```sql
SELECT DBUSERNAME, ACTION_NAME, EVENT_TIMESTAMP
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
WHERE TRUNC(EVENT_TIMESTAMP) = TRUNC(SYSDATE - 1)
ORDER BY EVENT_TIMESTAMP DESC;
```

**SQL attendu :**
```sql
SELECT DBUSERNAME, ACTION_NAME, EVENT_TIMESTAMP
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
ORDER BY EVENT_TIMESTAMP DESC
FETCH FIRST 1 ROWS ONLY;
```

**Ce qui ne va pas :**
- La question ne demande aucune période. "La toute dernière action" = le dernier enregistrement de toute la table, quel que soit le jour.
- Le modèle ajoute `TRUNC(EVENT_TIMESTAMP) = TRUNC(SYSDATE - 1)` (= hier), ce qui peut soit retourner aucun résultat (si la dernière action date d'aujourd'hui ou d'avant-hier), soit retourner des dizaines de lignes (toutes celles d'hier).
- `FETCH FIRST 1 ROWS ONLY` est absent, alors qu'on demande une seule et unique action.
- La requête s'exécute sans erreur Oracle, mais peut retourner 0 ligne ou beaucoup de lignes.

**Cause racine :**  
Le modèle a sur-appris un réflexe : dès qu'une question contient "dernière" ou "dernier", il ajoute automatiquement une condition temporelle. Ce comportement, correct pour "les derniers jours" ou "la dernière semaine", est inappropriate pour "la toute dernière action globale" qui n'a pas de contrainte de période.

**Impact utilisateur :** Faux négatif possible — le système répond "aucune activité" ou retourne une liste alors qu'on attendait une seule ligne.

---

### ❌ Variante V3 — "Donne la dernière action globale en base, sans filtrer sur hier."

**SQL généré par V11 :**
```sql
SELECT ID, EVENT_TIMESTAMP, DBUSERNAME, ACTION_NAME, OBJECT_NAME, EVENT_TYPE
FROM SMART2DSECU.UNIFIED_AUDIT_DATA
WHERE TRUNC(EVENT_TIMESTAMP) = TRUNC(SYSDATE - 1)
ORDER BY EVENT_TIMESTAMP DESC;
```

**Erreur Oracle :** `ORA-00904: "EVENT_TYPE": invalid identifier`

**Ce qui ne va pas :**
- Même erreur de réflexe que Q13 : filtre `SYSDATE-1` ajouté alors que la question le **interdit explicitement** ("sans filtrer sur hier").
- Pire : le modèle invente une colonne `EVENT_TYPE` qui n'existe pas dans la table. La liste des colonnes réelles ne contient pas `EVENT_TYPE` (la colonne correcte serait `ACTION_NAME` ou `AUDIT_TYPE`).
- `FETCH FIRST 1 ROWS ONLY` toujours absent.
- Cette variante est particulièrement révélatrice : même si la question précise explicitement de ne pas filtrer sur hier, le modèle le fait quand même.

**Cause racine :**  
Deux problèmes combinés : (1) réflexe `SYSDATE-1` inamovible sur les questions de "dernière action", et (2) hallucination de colonnes — le modèle invente des noms de colonnes plausibles (`EVENT_TYPE`) qui sonnent Oracle mais n'existent pas dans le schéma. Cela indique que le dataset d'entraînement ne renforce pas assez la liste des colonnes autorisées.

**Impact utilisateur :** Erreur Oracle bloquante. L'utilisateur reçoit un message d'erreur sur une colonne inexistante.

## 6. Recommandations pour un entraînement V12 ciblé

Les failles restantes sont précises et documentées. Voici les catégories d'exemples à ajouter en priorité dans le dataset :

### Priorité 1 — Mapping `"poste"` / `"machine"` / `"station"` → `USERHOST`
Aucune correction observée entre V10 et V11. Il faut ajouter des paires d'entraînement variées :
```
poste le plus actif → GROUP BY USERHOST
machine la plus connectée → GROUP BY USERHOST
station réseau → USERHOST
"sur quel poste" → WHERE USERHOST = ...
```
**Volume recommandé :** 20 à 30 paires avec formulations variées.

### Priorité 2 — "Dernière action globale" sans filtre de date
Le modèle sur-applique `SYSDATE-1`. Il faut des exemples où l'intention est clairement "toute la table, pas de période" :
```
toute dernière action → ORDER BY EVENT_TIMESTAMP DESC FETCH FIRST 1 ROWS ONLY
dernier événement enregistré → ORDER BY EVENT_TIMESTAMP DESC FETCH FIRST 1 ROWS ONLY
action la plus récente → ORDER BY EVENT_TIMESTAMP DESC FETCH FIRST 1 ROWS ONLY
```
**Volume recommandé :** 15 paires, sans aucun filtre WHERE sur les dates.

### Priorité 3 — `GROUP BY` systématique avec `COUNT`
Le modèle génère parfois `SELECT col, COUNT(*) ... ORDER BY COUNT(*)` sans le `GROUP BY` obligatoire. Il faut ancrer ce pattern :
```
tout exemple avec COUNT(*) et une colonne non agrégée doit avoir GROUP BY cette colonne
```
**Volume recommandé :** Vérifier et corriger tous les exemples du dataset qui omettent `GROUP BY`.

### Priorité 4 — Comptage utilisateurs dans les données d'audit
La question "combien d'utilisateurs" doit toujours produire `COUNT(DISTINCT DBUSERNAME)` sur `UNIFIED_AUDIT_DATA`, jamais `DBA_USERS` :
```
combien d'utilisateurs dans les traces → COUNT(DISTINCT DBUSERNAME) FROM SMART2DSECU.UNIFIED_AUDIT_DATA
nombre de comptes qui apparaissent dans l'audit → COUNT(DISTINCT DBUSERNAME)
```
**Volume recommandé :** 15 paires.

### Priorité 5 — Colonnes strictement autorisées
`EVENT_TYPE` est inventée. Le modèle doit être renforcé sur la liste exacte des colonnes :
`ID, AUDIT_TYPE, SESSIONID, OS_USERNAME, USERHOST, TERMINAL, AUTHENTICATION_TYPE, DBUSERNAME, CLIENT_PROGRAM_NAME, OBJECT_SCHEMA, OBJECT_NAME, SQL_TEXT, SQL_BINDS, EVENT_TIMESTAMP, ACTION_NAME, INSTANCE`

Aucune autre colonne ne doit apparaître. Le dataset doit couvrir toutes ces colonnes avec des exemples, en particulier `AUDIT_TYPE`, `SESSIONID`, `AUTHENTICATION_TYPE`, `CLIENT_PROGRAM_NAME` qui sont sous-représentées.

## 7. Verdict final

| | Score |
|---|---|
| Total réussi | **4 / 10 (40 %)** |
| Anciennes failles corrigées | **3 / 7 (42,86 %)** (objectif : 7/7) |
| Stabilité sur variantes proches | **1 / 3 (33,33 %)** |

Le LoRA V11 progresse sur les requêtes orientées `OBJECT_NAME` (top objets, top tables, modifications). C'est le gain principal de cet entraînement.

Les failles restantes sont toutes des problèmes de **biais dominant** : le modèle a trop souvent vu certains patterns pendant l'entraînement (`DBUSERNAME`, `SYSDATE-1`, `GROUP BY DBUSERNAME`) et ils prennent le dessus même quand la question pointe ailleurs.

Un V12 avec 60 à 80 exemples supplémentaires bien ciblés sur les 4 priorités ci-dessus devrait permettre de dépasser **75 % de réussite** sur ce benchmark.
